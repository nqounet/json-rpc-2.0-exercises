# 演習 NNN — <短いタイトル>

## 目的

- 実装すべき JSON-RPC 2.0 のメソッド（または振る舞い）と期待される動作を簡潔に説明してください。

## 仕様

### JSON-RPC フィールド

- `jsonrpc`: リクエスト・レスポンスともに文字列 `"2.0"` を使用すること。
- `method`: 実装対象のメソッド名（例: `"method_name"`）。
- `params`: 必要なパラメータの型や形式（位置引数／名前付き引数）を明記すること。
- `result`: 正常時に返すレスポンスの内容・型を明記すること。

### エラー処理

- 条件と対応するエラーコードを列挙する（可能なら標準の JSON-RPC エラーコードを利用すること）：
  - `-32700` Parse error: 無効な JSON
  - `-32600` Invalid Request: 有効な JSON-RPC オブジェクトでない
  - `-32601` Method not found: 未知のメソッド
  - `-32602` Invalid params: パラメータが欠落、または型が不正
  - `-32603` Internal error: 処理中のサーバ内部エラー
- 通知（`id` が無いリクエスト）はレスポンスを返さない事。

## 注意点 / エッジケース

- 実装者が考慮すべき重要なエッジケースを列挙すること（空配列、`null` の `id`、大きな数値、重複パラメータ、混合バッチなど）。
- 言語固有の制約や許容する仮定があれば記載すること。

## 例

### リクエスト

```json
{"jsonrpc":"2.0","method":"method_name","params":[/* ... */],"id":1}
```

### レスポンス

```json
{"jsonrpc":"2.0","result":/* ... */,"id":1}
```

## 受け入れ条件

- ソリューションは `tests/NNN-name/` 以下の `request-*.json` を読み取り、`expected-*.json` と一致する JSON を出力すること。
- 最低 3 件のテストペアを用意すること: `0001` (正常系), `0002` (エッジ), `0003` (無効/悪意)
- テストは `request-XXXX.json` / `expected-XXXX.json` 形式を使用すること。バッチ応答は配列で、順不同を許容する場合は `*.meta.json` に記載すること。

## テスト

- テストペアは `tests/NNN-name/` に配置すること。
- `request-0001.json` / `expected-0001.json` (正常系)
- `request-0002.json` / `expected-0002.json` (エッジケース)
- `request-0003.json` / `expected-0003.json` (無効/悪意のある入力 — 例: 無効な params, 無効 JSON, 存在しないメソッド)
- 追加のテスト（`0004` 以降）で通知、空バッチ、混合バッチ、`id` の特殊値 (`null`, 非プリミティブ, 重複) をカバーすることを推奨します。
