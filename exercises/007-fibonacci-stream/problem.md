# Exercise 007 — fibonacci-stream（日本語）

目的:
- JSON-RPC 2.0 の `fibStream` メソッドを実装する。少し変わった点として、通知（`id` がない）を使ったときの振る舞いに注目する。

要件:
- `jsonrpc` は `"2.0"`。
- `method` は `"fibStream"`。
- `params` はオブジェクトで、`start`（整数、デフォルト 0）と `count`（正の整数）が含まれる。
- リクエストに `id` がある場合は、`start` から始めて `count` 個のフィボナッチ数（0,1,1,2,3,...）を `result` として配列で返すこと。
- リクエストが通知（`id` がない）で来た場合は、サーバは即座に何も返さない（レスポンスを送らない）。ただし、サーバ内でログ出力や別の非同期処理を行う想定で問題文に明記すること（実装課題としては返答しないことが重要）。
- `start` や `count` が整数でない、または `count` <= 0 の場合は `-32602`（Invalid params）を返す。
- 大きな `count`（例えば 10000）に対しては、実装上の制約を明記してテスト側で範囲を限定すること（問題文に許容される最大 `count` を書く）。

受け入れ条件:
- `id` がある通常リクエストでは正しい配列を返す。
- 通知では一切のレスポンスを返さない（テストではレスポンスが存在しないことを期待する）。
- 無効な JSON には `-32700` を返し `id` は `null`。

追加注記:
- 仕様を少し変える意図で、通知と通常リクエストでサーバの副作用（ログ／キュー）を想定させるが、テストはレスポンスの有無と内容に限定する。

難易度: ⭐⭐

例:
```json
{"jsonrpc":"2.0","method":"fibStream","params":{"start":0,"count":6},"id":1}
```
期待:
```json
{"jsonrpc":"2.0","result":[0,1,1,2,3,5],"id":1}
```
