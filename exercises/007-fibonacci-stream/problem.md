# Exercise 007 — fibonacci-stream（日本語）

## Objective

- JSON-RPC 2.0 の `fibStream` メソッドを実装する。通知と通常リクエストでの振る舞いの違いに注意する。

## Specification

### JSON-RPC フィールド

- `jsonrpc`: リクエストとレスポンスで文字列 `"2.0"` とすること。
- `method`: `"fibStream"`。
- `params`: オブジェクトで、`start`（整数、デフォルト 0）と `count`（正の整数）を含む。
- `result`: `id` があるリクエストでは、`start` から始まる `count` 個のフィボナッチ数を配列で返す。

### 動作と検証

- `id` がある通常リクエストでは `start` から `count` 個のフィボナッチ数を配列で返す。
- 通知（`id` がない）ではレスポンスを送らない。
- `start` は整数 >= 0、`count` は正の整数であること。無効な場合は `-32602`（Invalid params）を返す。
- 実装はリソース枯渇を避けるために `count` の上限（例: 1000）を設け、問題文に明記すること。

## エラー処理

- 標準の JSON-RPC エラーコードを使用する（`-32700`、`-32600`、`-32601`、`-32602`、`-32603`）。
- 通知はレスポンスを返してはいけない。

## 注意事項 / エッジケース

- テストは通知がレスポンスを生成しないことと、通常リクエストが期待される配列を返すことを検証する。

## 例

Request:
```json
{"jsonrpc":"2.0","method":"fibStream","params":{"start":0,"count":6},"id":1}
```

期待:
```json
{"jsonrpc":"2.0","result":[0,1,1,2,3,5],"id":1}
```

## 受け入れ条件

- `id` がある通常リクエストでは正しい配列を返す。
- 通知ではレスポンスを返さない。
- 無効な JSON は `-32700` を返し `id` は `null`。
